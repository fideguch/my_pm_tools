# 新ツール導入規則

## 目的

新しい外部 API ツールを my_pm_tools に追加する際の品質ゲート。
「血液が通る状態」（end-to-end 接続成功）を保証するプロセス。

## 導入フロー（MANDATORY）

### Phase 0: 既存 MCP チェック

新ツール実装前に、対象サービスの MCP サーバーが既に存在しないか確認する。

1. **MCP レジストリ検索**: [MCP Hub](https://github.com/modelcontextprotocol) および npm/PyPI で `mcp-server-<service>` を検索
2. **既存 MCP がある場合**: カスタム実装が必要な理由を明記する
   - PM ワークフロー固有のデータ変換が必要
   - 既存 MCP にない操作（書き込み、一括処理等）が必要
   - パフォーマンス要件（バッチ処理、キャッシュ等）を満たせない
3. **既存 MCP がない場合**: Phase 1 に進む

### Phase 1: API 仕様理解（コード前）

1. **公式ドキュメント読了**: API リファレンス・ガイドを Context7 または WebFetch で取得
2. **認証方式の確認**: OAuth スコープ、API キー、サービスアカウント等
3. **レスポンス型の実物確認**: curl または node で実 API を叩き、レスポンスの形を確認
4. **エッジケースの特定**:
   - 空レスポンス（データなし）の形
   - エラーレスポンスの形（400, 403, 404, 429）
   - 日本語・特殊文字を含むパラメータの扱い
   - ページネーションの有無と形式

#### プロトコル別チェック

##### REST API

- レスポンスステータスコード一覧確認（特に 200, 201, 204, 400, 401, 403, 404, 429）
- クエリパラメータ/リクエストボディの形式確認（JSON, form-urlencoded 等）
- HTTP 204 No Content の処理パターン（空レスポンスのパース回避）

##### GraphQL API

- エラー形式: HTTP 200 + `errors` 配列パターン（ステータスコードでの判定不可）
- Complexity/Rate limit のクエリ単位制限（クエリコスト計算）
- Relay-style cursor pagination（`first`/`after`/`pageInfo`）
- フラグメント再利用戦略（共通フィールドの DRY 化）
- codegen / SDK 利用可否の判断（型安全性 vs 柔軟性のトレードオフ）

### Phase 2: インターフェース設計

1. **型定義**: `src/types/workspace.ts` に readonly 型を追加
2. **GoogleClient 拡張**: インターフェースにメソッド追加
3. **メタデータ取得メソッド**: 動的情報（シート名、フォルダ名等）を事前取得する手段を必ず用意
4. **クエリビルダー**: ユーザー入力を API 仕様に準拠した形に自動変換
   - 例: `searchDrive('test')` → `name contains 'test' and trashed = false`

### Phase 3: 実装

1. **ツール実装**: `src/tools/<category>/<name>.ts`
2. **Zod スキーマ**: `src/schemas/index.ts`
3. **ツール登録**: `src/tools/index.ts` にアノテーション付きで登録
4. **認証方式分岐**: サービスに応じた認証フォールバックを実装

   | 認証方式     | フォールバック                   | 例               |
   | ------------ | -------------------------------- | ---------------- |
   | Google OAuth | ADC → env vars → エラー（3段階） | Google Workspace |
   | API Key      | env var → エラー（2段階）        | Linear, Notion   |
   | OAuth 2.0    | token → refresh → 再認証要求     | GitHub           |

#### 破壊的操作の安全設計

- `destructiveHint: true` アノテーション必須（MCP ツール登録時）
- 副作用パラメータ（メール送信、通知等）のデフォルト = 無効（`none`）
- PATCH vs PUT の選択: 部分更新は PATCH 推奨（意図しないフィールドクリアを防止）
- HTTP 204 対応: `googleFetchCore` に空レスポンス分岐を追加（`response.text()` で空文字チェック）

#### クライアント設計判断

- **既存クライアント拡張**: 同じサービスプロバイダの新 API（例: Google Calendar Write → `GoogleClient` に追加）
- **新規クライアント作成**: 新しいサービスプロバイダ（例: Linear → `LinearClient` を新規作成）
- **命名**: `src/utils/<service>-client.ts`、DI パターン準拠

### Phase 4: テスト（3 層）

1. **ユニットテスト**: モック使用、正常系 + エラー系 + エッジケース
2. **モック統合テスト**: `tests/live/` で read→write→rollback サイクル
3. **ライブテスト**: `GOOGLE_LIVE_TEST=1` ゲート付き、実 API 接続確認

### Phase 5: 接続検証（血液検査）

以下が全て成功するまで「導入完了」としない:

- [ ] モックテスト全パス
- [ ] ライブテストで実 API からデータ取得成功
- [ ] 書き込みテストでロールバック成功
- [ ] 日本語パラメータで動作確認
- [ ] 空データ/エラーケースでクラッシュしない
